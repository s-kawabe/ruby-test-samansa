# サブスクリプション管理システムの実装

動画配信サービスにおける定期購読の開始・更新・解約を管理する API を Ruby on Rails で実装してください

実務に耐えうる設計・実装（拡張性・分析可能性・冪等性・スケーラビリティ等）を意識し、README に設計の概要や工夫したポイントを記載してください（日本語もしくは英語）

不明点は適切に定義して構いません

- ユーザは Apple のアプリ内課金でサブスクリプションを購入する
- 決済完了後、アプリ側から Rails API に決済情報を送信し、サブスクリプションを仮開始する
- その後、Apple からの Webhook 通知（開始・更新・解約）を受信して、サブスクリプションの状態を更新する。署名検証は省略して良い
- 解約時でも現在の有効期限までは利用可能とする

評価の観点：要件定義力・設計力・拡張性・説明力

## クライアント -> サーバ (決済完了直後)

```json
{
  "user_id": "string",
  "transaction_id": "string",
  "product_id": "string"
}
```

- user_id: 本来は Cookie 等から取得する値だが、今回わかりやすくするためパラメータとして渡す形にする。検証不要
- transaction_id: サブスクリプションを一意に識別する ID。同じサブスクリプションなら自動更新されても同じ値
- product_id: サブスクリプションプランの ID。例：com.samansa.subscription.monthly

この時点では仮開始。Apple Webhook 到着で本開始。仮開始中は視聴不可。

## Apple -> サーバ （Webhook）

```json
{
  "notification_uuid": "string",
  "type": "PURCHASE" "RENEW" "CANCEL",
  "transaction_id": "string",
  "product_id": "string",
  "amount": "3.9",
  "currency": "USD",
  "purchase_date": "2025-10-01T12:00:00Z",
  "expires_date": "2025-11-01T12:00:00Z",
}
```

- notification_uuid: 通知ごとに一意の値
- type: 通知の種類。PURCHASE は新規購入、RENEW は自動更新、CANCEL は解約
- transaction_id: サブスクリプションを一意に識別する ID。同じサブスクリプションなら自動更新されても同じ値
- product_id: サブスクリプションプランの ID。例：com.samansa.subscription.monthly
- amount / currency: 課金金額と通貨
- purchase_date: 現在のサブスクリプション期間の開始日時
- expires_date: 次回更新またはサブスクリプション終了日時

## 工夫したポイント

### 1. 通知の到着順序に依存しない冪等な状態遷移

モバイル回線の不安定さにより、Apple からのサーバー間通知がクライアントのリクエストより先に届くことが本番で実際に起きる。「どちらが先に届いても最終状態が同じになる」よう Upsert で設計し、すでに有効化済みのサブスクリプションを仮状態に戻さないガードを入れた。

### 2. DB 制約による Webhook 重複排除

通知ごとのユニーク ID に UNIQUE 制約を設けることで、Apple の再送による重複通知をアプリコードではなくデータ層で弾く。状態チェックのロジックを書かずに二重処理を防げるため、処理漏れ・二重更新の両方に対してシンプルに対処できる。

### 3. Webhook 受付と処理の分離（スループット・タイムアウト対策）

Webhook 受信 API は通知の記録とジョブのエンキューのみ行い、即座に 200 OK を返す。重い処理はバックグラウンドで非同期に捌く。Apple は一定時間内に応答がないと再送するため、同期処理にするとタイムアウト起点の重複が増える。またセール時の大量通知でも API が詰まらない構造になる。

### 4. 有効期限切れを動的に判定（stale な状態を持たない）

「期限切れ」をDBに保存せず、有効期限と現在時刻の比較で動的に導出する。定期バッチでステータスを更新する設計にすると、バッチ遅延やスキップで実態とDBの状態が乖離する。原理的にその不整合が起きない設計にするためこうした。

### 5. 課金履歴を不変ログとして分離（分析可能性）

契約の最新状態と課金イベントの履歴を別テーブルに持ち、履歴は追記のみにした。最新状態は更新で上書きされるが、全課金イベントの時系列は失われない。MRR・チャーン率・平均継続期間などのビジネス分析に使える形式を最初から確保している。

### 6. 課金ストアをレコードに記録（拡張性）

現状は Apple のみだが、Google Play 等への将来対応を想定してストア種別をレコードに持たせた。ストアごとにペイロード形式や処理ロジックが異なるため、追加時の影響範囲を局所化できる。

### 7. 視聴権限確認 API の追加（要件外）

要件には含まれていないが、動画配信サービスとして必須と判断して追加した。視聴可否は動的に計算して `viewable` フラグで返すため、判定ロジックをサーバー側に閉じ込めクライアントを薄く保てる。`status` と有効期限も合わせて返すことで、決済処理中・期限切れ等の状態に応じたクライアントの UI 分岐にも対応できる。
