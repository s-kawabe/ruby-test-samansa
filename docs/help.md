## Q1. AppleのWebhookが届くのに、クライアント向けの仮開始APIが必要な理由は？

**結論：今回のスペック範囲では実質不要。ただし実務上の意味がある。**

### 仮開始APIがあっても視聴体験は変わらない

「仮開始中は視聴不可」なので、ユーザーが実際にコンテンツを視聴できるタイミングはどちらの設計でも変わらない。

| | 仮開始APIあり | 仮開始APIなし |
|--|--|--|
| 視聴できるタイミング | Webhook到着後 | Webhook到着後 |
| 実質的な差 | なし | なし |

### それでも実務で存在する理由

1. **運用・サポートの可視性**
   - 「決済完了したがWebhookがまだ来ていない」状態をDBで追跡できる
   - Webhookが永久に届かない障害を検知するためのレコードになる

2. **本来はここでレシート検証を行う（今回はスコープ外）**
   - 実際のプロダクトではAppleのレシート検証APIを叩いてtransaction_idの正当性を確認する
   - その処理のフックポイントとして仮開始APIが存在する
   - 詳細は [Q3. レシート検証とは何を確認するのか？](#q3-レシート検証とは何を確認するのか) を参照

3. **マルチデバイス対応**
   - 別デバイスから購読状態を確認するとき、`provisional` レコードがあれば「処理中」と返せる

### 本プロジェクトでの扱い

問題文で明示的に要求されているため実装する。
README にはこの設計判断の理由を記載することで「要件定義力・説明力」の評価につなげる。

## Q2. `expired` バッチ処理と視聴権限確認APIは必要か？

### `expired` バッチ処理は不要

バッチでDBステータスを `expired` に更新する方式には本質的な問題がある。

```
expires_date: 2026-04-17 00:00:00
バッチ実行:   2026-04-17 01:00:00
```

この1時間、DBは `cancelled` だが実態は期限切れという**不整合が常に存在する**。課金系の視聴権限判定にラグのあるバッチを使うのは精度が下がる。

**採用する方式：動的判定**

```sql
-- 視聴可否は毎回この条件で判定する
status IN ('active', 'cancelled') AND expires_date > NOW()
```

`expired` はDBに書き込むステータスではなく、`expires_date` と現在時刻の比較で導出する値として扱う。
これにより常に正確な判定が可能で、バッチの運用コスト・遅延・失敗リスクもゼロになる。

### 視聴権限確認APIは要件の抜け

**現在の要件定義書には記載がないが、動画配信サービスとして実質必須のAPI。**

```
ユーザーが動画再生ボタンを押す
    ↓
GET /api/v1/users/:user_id/subscription/viewable
    ↓
{ "viewable": true, "expires_at": "2026-05-01T00:00:00Z" }
    ↓
true → 再生 / false → 課金導線へ
```

このAPIがなければアプリ側がサブスクリプションの有効性を判断できない。

**要件定義書に追加すべき項目：**

| フィールド | 内容 |
|--|--|
| エンドポイント | `GET /api/v1/users/:user_id/subscription` |
| レスポンス | `viewable`（boolean）、`status`、`expires_at` |
| 判定ロジック | `status IN ('active', 'cancelled') AND expires_date > NOW()` |

### まとめ

| 項目 | 判断 |
|------|------|
| `expired` バッチ処理 | 不要（動的判定で代替） |
| `expired` をDBステータスとして保存 | 不要（`expires_date` から導出） |
| 視聴権限確認API | 要件定義書に追加する |

## Q3. レシート検証とは何を確認するのか？

**一言でいうと：「このtransaction_idは本当にAppleが発行したものか・このアプリ向けのものか・使い回しではないか」を確認すること。**

### なぜ検証が必要か

クライアント（iOSアプリ）からサーバーに送られてくる `transaction_id` は、あくまでクライアントが「これが購入IDだ」と主張しているだけに過ぎない。
悪意あるユーザーが偽のIDや他人の購入IDを送り込んでくる可能性があるため、Appleのサーバーに直接問い合わせて「本物かどうか」を確かめる必要がある。

```
[悪意あるクライアント]
  → 偽の transaction_id を送信
  → サーバーがAppleに確認せず信用すると無料で視聴できてしまう
```

### 具体的に何を確かめるか

AppleのStoreKit 2では、サーバー側から以下のエンドポイントに `transaction_id` を渡して照合する。

```
GET https://api.storekit.itunes.apple.com/inApps/v2/transactions/{transactionId}
```

このレスポンスに含まれるJWT（JWS形式）をデコードして、次の4点を確認する。

| 確認項目 | 内容 | 不正の例 |
|---|---|---|
| **署名の正当性** | AppleのCA証明書で署名されたJWTか | 自分で作ったJWTを送り込む |
| **bundleId の一致** | このアプリ向けの購入か | 別アプリで購入したレシートを使い回す |
| **productId の一致** | 購入したプランが正しいか | 安いプランのIDで高いプランを要求する |
| **environment の確認** | Sandbox購入をProduction環境で使っていないか | テスト購入で本番の権限を得る |

### replay攻撃への対策

`transaction_id` は1回の購入に対してAppleが発行する**一意のID**。
同じIDが複数ユーザーのアカウントに紐づいていたら「使い回し」＝不正と判断できる。

```sql
-- 同じtransaction_idが別ユーザーに存在しないかを確認する
SELECT COUNT(*) FROM subscriptions
WHERE transaction_id = :transaction_id
  AND user_id != :user_id
-- 0件でなければ不正
```

### WebhookとのID照合

本プロジェクトでは「仮開始API → Webhook受信 → 有効化」という2段階フローを採用している。
レシート検証は仮開始APIの時点で行うことで、その後に届くWebhookとIDを照合しやすくなる。

```
[仮開始API]
  transaction_id を受け取る
      ↓
  Appleに問い合わせ → bundleId / productId / 署名 / replay を検証
      ↓
  DBに provisional レコードを作成（transaction_id を保存）

[Webhookが届いたとき]
  AppleからのJWT内の transaction_id
      ↓
  DBの provisionalレコードと照合 → 一致すれば active に更新
```

### まとめ

| 確認すること | 防げる不正 |
|---|---|
| 署名の正当性 | 偽造されたレシートの使用 |
| bundleId の一致 | 別アプリの購入の使い回し |
| productId の一致 | プランの改ざん |
| environment の確認 | Sandbox購入による本番権限取得 |
| replay攻撃チェック | 1つの購入IDの複数人への流用 |
