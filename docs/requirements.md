# 要件定義書

## Purpose

動画配信サービスにおいて、Apple のアプリ内課金を起点としたサブスクリプションのライフサイクル（仮開始・本開始・更新・解約）を正確かつ冪等に管理する。ユーザーが決済を完了した時点から、Apple Webhook を通じた状態遷移・有効期限管理までを一貫して処理する API を提供する。実務に耐えうる設計（拡張性・分析可能性・冪等性・スケーラビリティ）を実現することを目的とする。

### 対象範囲
- クライアント（アプリ）からの決済完了通知の受信と仮開始処理
- Apple Webhook 通知（PURCHASE / RENEW / CANCEL）の受信と状態更新
- サブスクリプションの有効期限管理（解約後も期限まで利用可能）

### 対象外

- **Apple 署名検証** — 本コーディングテストのスコープ外として省略する。実際のプロダクトでは Apple の公開鍵を用いた JWT 署名検証が必須となる
- **ユーザー認証・検証** — 本コーディングテストのスコープ外として省略する。実際のプロダクトでは Cookie やトークンを通じてユーザーを識別・検証する必要がある

---

## Requirement: サブスクリプション仮開始

クライアントからの決済完了通知を受信し、仮開始状態でサブスクリプションを登録する。

**ユーザーストーリー**: ユーザーとして、アプリ内課金完了直後にサーバーへ通知を送りたい。なぜなら Apple Webhook 到着前の状態を記録し、処理の追跡を可能にしたいからだ。

### Scenario: 正常登録

- **GIVEN** 当該サブスクリプションに紐づく登録がまだ存在しない
- **WHEN** 決済完了通知がクライアントから送信される
- **THEN** サブスクリプションを仮開始状態で登録し、正常受付を返す

### Scenario: Webhook 先着済みの場合

- **GIVEN** 同一サブスクリプションがすでに有効状態である
- **WHEN** 決済完了通知がクライアントから送信される
- **THEN** 有効化済みの状態をそのまま維持し、降格させない

### Scenario: 仮開始中の視聴

- **GIVEN** ユーザーのサブスクリプションが仮開始状態である
- **WHEN** ユーザーが視聴を試みる
- **THEN** 視聴を許可しない

---

## Requirement: サブスクリプション本開始

Apple Webhook の購入確定通知（PURCHASE）を受信し、対象サブスクリプションを有効状態に遷移させる。

**ユーザーストーリー**: システムとして、Apple からの購入確定通知を受け取りたい。なぜなら仮開始状態を本開始に昇格させ、ユーザーの視聴権限を有効化したいからだ。

#### Scenario: 正常な本開始

- **GIVEN** 対応するサブスクリプションが仮開始状態で存在する
- **WHEN** 購入確定通知（PURCHASE）が届く
- **THEN** サブスクリプションを有効状態に遷移させ、有効期限をセットする

#### Scenario: 仮開始レコードが存在しない場合

- **GIVEN** 対応するサブスクリプションが仮開始状態では存在しない
- **WHEN** 購入確定通知（PURCHASE）が届く
- **THEN** 新規に有効状態のサブスクリプションを作成する（Webhook 先着ケース）

---

## Requirement: サブスクリプション識別

一つのサブスクリプションを自動更新をまたいで一意に識別し、PURCHASE・RENEW・CANCEL のすべての通知にわたって同一サブスクリプションとして処理する。

**ユーザーストーリー**: システムとして、複数の Webhook にわたって同一サブスクリプションを特定したい。なぜなら正確な状態遷移を保証するためにはサブスクリプションの識別が一貫している必要があるからだ。

#### Scenario: 自動更新時の識別

- **GIVEN** 初回購入により有効なサブスクリプションがすでに存在する
- **WHEN** 自動更新通知（RENEW）が届く
- **THEN** 初回購入時と同一のサブスクリプションとして識別し、状態を更新する

---

## Requirement: 有効期限更新

Apple Webhook の自動更新通知（RENEW）を受信し、対象サブスクリプションの有効期限を延長する。

**ユーザーストーリー**: システムとして、自動更新の通知を受け取りたい。なぜなら有効期限を延長し、ユーザーの継続視聴権限を維持したいからだ。

#### Scenario: 正常な更新

- **GIVEN** 対象サブスクリプションが有効状態で存在する
- **WHEN** 自動更新通知（RENEW）が届く
- **THEN** 有効期限を新しい値に延長し、更新イベントを記録する

---

## Requirement: 解約処理

Apple Webhook の解約通知（CANCEL）を受信し、サブスクリプションを解約済み状態に遷移させる。

**ユーザーストーリー**: システムとして、解約通知を受け取りたい。なぜなら解約後も有効期限まではユーザーの視聴権限を維持しつつ、解約済みであることを記録したいからだ。

#### Scenario: 正常な解約

- **GIVEN** 対象サブスクリプションが有効状態で存在する
- **WHEN** 解約通知（CANCEL）が届く
- **THEN** サブスクリプションを解約済み状態に遷移させ、有効期限は変更しない

#### Scenario: 解約後の視聴権限（有効期限内）

- **GIVEN** サブスクリプションが解約済みであり、かつ有効期限内である
- **WHEN** ユーザーが視聴を試みる
- **THEN** 視聴を許可する

#### Scenario: 解約後の期限切れ

- **GIVEN** サブスクリプションが解約済みであり、かつ有効期限を過ぎている
- **WHEN** ユーザーが視聴を試みる
- **THEN** 視聴を許可しない

## 非機能要件

### 冪等性

- 同一通知の重複受信に対して、システムは安全にスキップし二重処理しないこと
- ネットワーク障害等による再送を安全に受け入れられること

### 分析可能性

- 課金金額・通貨・購入日時・有効期限を保持し、後から収益分析・チャーン分析が可能なこと
- **ライフサイクル上の主要イベント**（購入確定・更新・解約など）が時系列で追えること。監査やファネル分析に使えること
- 収益・チャーンなどの分析において、**期間・プラン・課金ストア** などの軸でセグメントした集計ができること。分析軸が増えた場合も、保持しているデータから再集計・拡張できること

### 拡張性

- 現状は Apple に限定するが、将来 **別の課金ストア**（例: Google Play）からの通知を追加しても、既存のサブスクリプション状態モデルと矛盾しないこと
- **新しい通知種別** や課金ライフサイクル上のイベントが増えた場合、既存データの意味を壊さずに取り込める余地があること
- **サブスクリプションプランの追加・変更**（プラン識別子の増減、価格・通貨の変更）に追随できること。既存ユーザーの履歴と整合する
- クライアントや運用ツール向けの **読み取り API** を増やしても、コアとなる課金状態の単一の意味論を保てること

### パフォーマンス

- Webhook の急増時でも処理が詰まらないよう、非同期処理を考慮すること
